# v0.1.4 Run Safety Path Plan

Date: 2026-06-25

## Product Intent

v0.1.4 adds an opt-in argv/non-shell execution path for local CLI `run` while
preserving the existing explicit shell command behavior. The goal is to let a
human caller pass a program and arguments as a structured argv list so
Lab-Sidecar can execute them with `subprocess.Popen(argv, shell=False)` and
record the structure in task-local reproduce metadata.

The existing primary path remains:

```text
run / ingest -> collect -> figures -> report -> slides
```

This is a run interface hardening slice, not a product-surface expansion.

## Non-Goals

- No Web UI, FastAPI or HTTP service, hosted service behavior, cloud sync, or
  remote runner.
- No OS sandboxing, container runtime, malware detection, shell interception
  layer, or command policy engine.
- No default AI analysis, autonomous research conclusions, statistical
  significance, model superiority claims, paper conclusions, or deployment
  advice.
- No MCP/V2 product-surface expansion. MCP-facing command safety remains a
  separate experimental local adapter boundary.
- No safe profiles in this slice. `safe_profile` is reserved in metadata for a
  later narrow design.
- No breaking change to `labsidecar run "<command>"`.

## RunSpec Design

Introduce an internal `RunSpec` model that can represent both execution modes:

```text
mode: shell | argv
command_text: str
argv: list[str] | null
safe_profile: str | null
```

Compatibility rules:

- For legacy shell mode, `command_text` is exactly the user-provided command
  string and `argv` is `null`.
- For argv mode, `argv` is the exact list passed to `Popen(..., shell=False)`.
  `command_text` is a human-readable preview built with shell-style quoting
  for display and `reproduce/command.txt`; it is not re-parsed for execution.
- `TaskRecord.command` remains present and continues to contain the display
  command string for both modes. Existing consumers that read `command` keep
  working.
- Additive top-level task metadata records `run_mode`, `argv`, and
  `safe_profile` so `manifest.json` can distinguish shell and argv runs without
  changing `schema_version`.
- Add `reproduce/run.json` as the structured reproduce file with the same
  fields plus a short note that argv mode is non-shell execution, not
  sandboxing.

## CLI Syntax Choice

Use:

```bash
labsidecar run --no-shell -- python examples/simple-success/train.py --output metrics.csv
```

Reasoning:

- `--no-shell` is explicit and describes the execution distinction directly.
- `--` is the standard delimiter that lets child options such as `--output`
  belong to the argv command instead of the Lab-Sidecar CLI.
- Typer/Click can support this with command-level extra args. Tests must cover
  child options, arguments with spaces, and shell metacharacters.

Legacy shell mode remains:

```bash
labsidecar run "python examples/simple-success/train.py --output metrics.csv"
```

If `--no-shell` is provided without argv, the CLI should fail before creating a
task and tell the user to pass argv after `--`.

## Manifest, Reproduce, Traceability, And Package Impact

`manifest.json`:

- Keep `command` as a display string.
- Add optional `run_mode`, `argv`, and `safe_profile`.
- Do not rename artifact ids or path fields.
- Do not change `schema_version`.

`reproduce/command.txt`:

- Continue to contain a human-readable command preview.
- For shell mode, this remains the original command string.
- For argv mode, this is a quoted preview for humans, not an execution source.

`reproduce/run.json`:

- New structured metadata artifact for run tasks.
- Records `schema_version`, `run_mode`, `command_text`, `argv`, `safe_profile`,
  and a bounded note about non-shell execution.

Traceability:

- Add `run_mode`, `argv`, `safe_profile`, and `run_spec_path` to the task
  section.
- Include `reproduce/run.json` in traceability source paths when present.
- Keep traceability reference-only and bounded; no full logs, raw source
  bodies, worker prompt/response bodies, or artifact bytes.

Package:

- Add `reproduce/run.json` to the single-task package allowlist.
- Keep existing default omissions unchanged: full stdout/stderr logs, raw
  source files and refs, SQLite, worker audit files, sandbox files, and
  unrelated workspace files are still omitted by default.
- Package README and package summary may show bounded command preview/run mode,
  but not full logs or artifact bodies.

## Foreground And Background Behavior

Foreground shell mode:

- Continue `subprocess.Popen(command_text, shell=True, ...)`.
- Do not parse or normalize legacy command strings.

Foreground argv mode:

- Execute `subprocess.Popen(argv, shell=False, ...)`.
- Preserve each argv item exactly as passed by the CLI.

Background shell mode:

- Continue launching the internal worker and executing the shell command through
  the same shell path as today.

Background argv mode:

- Pass structured run spec metadata to `lab_sidecar.runner.worker` as JSON.
- Worker must call the same execution function with the same `RunSpec`, so argv
  structure is not flattened into a shell command.

## Cancellation And Stale Recovery Impact

No new lifecycle model is introduced. The existing PID fields, process-group
creation helpers, `cancel`, `refresh`, and stale-worker recovery remain the
authority for task lifecycle.

Expected effects:

- Cancellation targets the recorded child process or worker process exactly as
  before.
- Stale recovery continues to mark a manifest as failed when no active child or
  worker process can be verified.
- Tests should include at least one argv background cancellation or stale
  recovery path when stable on the local platform.

## Test Matrix

- Legacy shell foreground success still works and keeps the old `command`
  manifest field.
- Legacy shell foreground failure still records failure diagnostics.
- Argv foreground success writes stdout/stderr, manifest `run_mode: argv`,
  exact `argv`, `reproduce/run.json`, and `reproduce/command.txt`.
- Argv foreground failure returns a failed task with bounded diagnostics.
- Argv background success refreshes to completed without losing argv.
- Argv background cancellation or stale recovery preserves normal lifecycle
  behavior.
- Argv with spaces and shell metacharacters remains literal and does not invoke
  shell chaining, globbing, or variable expansion.
- `status`, `summarize`, `artifacts`, `package`, and traceability show or
  record run mode without embedding full logs or raw artifacts.
- Package omission contracts remain unchanged.
- Missing argv for `--no-shell` produces a clear CLI error and no task
  directory.

## Acceptance Matrix

| Area | Acceptance |
| --- | --- |
| Legacy compatibility | Existing `labsidecar run "<command>"` tests pass unchanged. |
| Argv execution | Foreground and background argv tasks use `shell=False` and preserve argv. |
| Metadata | Manifest, `reproduce/run.json`, traceability, and package metadata distinguish `shell` and `argv`. |
| Boundedness | Packages and traceability still omit full logs, raw sources, SQLite, worker audit bodies, sandbox files, and unrelated workspace files by default. |
| Docs | README, CLI spec, changelog, plan, and acceptance docs distinguish shell mode, argv/non-shell mode, MCP/V2 gates, and non-sandbox limits. |
| Validation | Focused tests, full pytest, Ruff, build, and wheel smoke are run and recorded. |

## Risks And Rollback Strategy

Risks:

- Typer parsing can accidentally consume child options or lose argv structure.
- Background worker serialization can flatten argv into a shell string.
- Metadata additions can accidentally break package or traceability contract
  tests.
- Wording can overclaim safety.

Mitigations:

- Require `--` in documented argv examples and test child options.
- Serialize `RunSpec` JSON to the worker rather than joining argv.
- Keep `command` and existing artifact ids intact.
- Limit docs to "argv/non-shell execution"; do not say sandbox.

Rollback:

- Because the feature is additive, rollback can remove the `--no-shell` CLI
  option and `reproduce/run.json` artifact while leaving legacy shell mode and
  existing manifests intact.
- Existing legacy task manifests remain readable because old fields are not
  renamed or removed.
