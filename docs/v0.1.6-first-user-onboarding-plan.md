# v0.1.6 First-User Onboarding And Recipe Trust Plan

Date: 2026-06-25

## Product Intent

v0.1.6 makes the public-alpha release line easier to trust for a stranger who
only has GitHub, the README, and repository docs. The goal is not to widen the
product surface. The goal is to make a first user understand, within about ten
minutes, how to install a released wheel, get the example inputs, run a demo,
generate shareable artifacts, verify the package, and understand the current
boundaries.

The user-facing path remains:

```text
run / ingest -> collect -> figures -> report -> slides -> validate -> package -> package-verify
```

Saved comparisons remain a bounded descriptive extension:

```text
run / ingest -> collect -> compare --save -> validate-comparison -> package-comparison -> package-verify
```

## Non-Goals

- No Web UI, FastAPI or HTTP service, hosted behavior, cloud sync, or remote
  runner.
- No default AI analysis, AI-authored conclusions, statistical significance,
  model superiority, paper conclusions, or deployment recommendations.
- No MCP/V2 product-surface expansion and no MCP schema work unless existing
  docs links or tests are broken.
- No TensorBoard, MLflow, JSONL, new collector family, broad recursive scan, or
  general experiment-tracking dashboard.
- No OS sandboxing claim for CLI `run`; it remains user-explicit local command
  execution.
- No v0.1.6 tag, GitHub release, or PyPI publish in this implementation slice.

## First-User Path

The canonical first-user path should be one clear story:

1. Download the release wheel from the GitHub release page.
2. Download the matching source archive or clone the repo only to get
   `examples/`.
3. Create and activate a virtual environment.
4. Install the wheel locally.
5. Run `labsidecar --help` and `labsidecar doctor`.
6. Run `examples/simple-success` through `run`.
7. Run `collect`, `figures`, `report`, `slides`, and `validate`.
8. Run `package` and `package-verify`.
9. Inspect the task directory and package output.

The docs should state that the wheel is the install-smoked artifact. Source
archives or a clone provide examples and docs; they are not the default
installed runtime surface. PyPI is not the default install promise for the
current public-alpha line unless maintainers separately publish and announce it.

## Recipe Gallery Design

`docs/recipes.md` will provide a compact recipe gallery rather than burying
every scenario in the README. Each recipe must include:

- target user scenario
- command sequence
- expected output and artifact categories
- next step
- common failures and fixes
- explicit limits: no AI conclusion, no statistical significance, and no model
  superiority judgment

Required recipes:

1. Local training run:
   `run -> collect -> figures -> report -> slides -> validate -> package -> package-verify`.
2. Ingest messy CSV or JSON results:
   `ingest examples/messy-csv-results` or
   `ingest examples/json-benchmark-results -> collect --config -> figures/report/slides/validate`.
3. Compare two local tasks and package the comparison:
   create or ingest two collected tasks, then
   `compare --save --figures --report -> validate-comparison -> package-comparison -> package-verify`.

## GitHub Release Asset Smoke Design

The release-trust story has two layers:

- A real published GitHub release asset smoke for v0.1.5. This is a manual,
  networked verification that installs the exact released wheel URL and runs
  the compact installed CLI path through metrics validation.
- A local v0.1.6 built-wheel smoke. This verifies the current repository build
  through installed `labsidecar`, generated figures, reports, slides,
  validation, task package verification, saved comparison generation, saved
  comparison validation, comparison packaging, and package verification.

Normal tests must not require GitHub or PyPI network access. HTTP(S) wheel URLs
are manual release verification inputs for `scripts/release_asset_smoke.py`.

## README And Docs Navigation Strategy

The README should become a clearer front door:

- Keep the first screen focused on what Lab-Sidecar is and what it is not.
- Point users to the first-user quickstart and recipe gallery near the install
  section.
- Keep detailed traceability, fallback chart, comparison, MCP, and maintainer
  release-smoke material in linked docs instead of expanding the README.
- Group docs as "Start here", "Recipes", "Reference", "Advanced local MCP",
  and "Maintainer release docs".
- Keep the CLI-first, file-first, local-first boundary visible.

Existing docs can remain, but the release install story must no longer compete
with an editable-install-only quickstart.

  URLs ending in `.whl` and rejects non-wheel HTTP(S) URLs

## Acceptance Matrix

| Area | Acceptance signal |
| --- | --- |
| First-user quickstart | A new user can follow the release wheel path, get examples, run the demo, validate, package, and verify the package. |
| Recipe gallery | At least three recipes cover local run, explicit CSV/JSON ingest, and saved comparison packaging. |
| Scope boundaries | Docs repeat no Web UI, HTTP service, hosted/cloud, remote runner, default AI analysis, statistical significance, or model-superiority claims. |
| Release asset trust | v0.1.5 published wheel URL smoke is recorded honestly; local v0.1.6 wheel and local release-asset smokes pass or failures are explained. |
| Versioning | `pyproject.toml`, `lab_sidecar/__init__.py`, and `CHANGELOG.md` agree on `0.1.6`. |
| Repo hygiene | No root `.lab-sidecar/`, generated package output, old dist artifacts, cache, or virtualenv is staged. |
| Release boundary | No v0.1.6 tag, GitHub release, or PyPI publish is created. |

## Risks And Rollback Strategy

- Risk: docs imply the wheel includes examples. Mitigation: state that examples
  come from the matching source archive or clone.
- Risk: docs overclaim PyPI availability. Mitigation: state PyPI is not the
  default public-alpha install promise.
- Risk: recipe commands drift from actual CLI behavior. Mitigation: base
  recipes on checked-in fixtures and validate with existing smoke tests.
- Risk: network smoke fails for external reasons. Mitigation: record the exact
  failure separately from product failures and keep default tests offline.
- Risk: README becomes long again. Mitigation: move details to quickstart,
  recipes, release checklist, and acceptance docs.

Rollback is documentation-only plus version metadata unless a small release
smoke test guard is added. Reverting the v0.1.6 docs and version bump restores
the v0.1.5 state without changing task artifact behavior.

## Release Boundary

This phase does not create a v0.1.6 tag, create or publish a GitHub release, or
publish to PyPI. The end state is code/docs committed on a branch, pushed, and
opened as a ready pull request for maintainer review.
