# v0.1.5 Install And Release Trust Plan

Date: 2026-06-25

## Product Intent

v0.1.5 is a public-alpha release hardening slice for install and release trust.
It should make it easier for a stranger to install Lab-Sidecar from GitHub
release artifacts, verify the CLI locally, and understand the current trust
boundary before running the local artifact workflow.

The release owner should also have a repeatable pre-release check that catches
version mismatches, stale distribution files, whitespace errors, accidental
generated state, and wrong release assets before a tag or GitHub release is
created.

The main product path remains:

```text
run / ingest -> collect -> figures -> report -> slides
```

## Non-Goals

- No Web UI, FastAPI/HTTP service, hosted service, cloud sync, remote runner,
  or multi-tenant service behavior.
- No default AI analysis, AI-authored research conclusions, statistical
  significance, model superiority, deployment advice, or paper conclusions.
- No MCP/V2 product-surface expansion.
- No broad artifact schema rewrite or generated `.lab-sidecar/` snapshot
  committed to the repository.
- No tag creation, GitHub release creation, PyPI upload, or package publishing
  in this implementation slice.

## Release And Install Trust Scope

This slice covers trust checks around the repository state and install path:

- Version consistency across `pyproject.toml`, `lab_sidecar/__init__.py`, and
  `CHANGELOG.md`.
- Local distribution artifact naming for the target version.
- SHA-256 digest reporting for built wheel and sdist files.
- Guardrails against stale v0.1.4 artifacts being treated as v0.1.5 artifacts.
- Repository hygiene checks for whitespace errors and accidental root
  `.lab-sidecar/` state.
- A smoke path that can install either a local wheel file or an explicitly
  supplied wheel URL without making the default test suite depend on network
  access.
- Documentation that positions GitHub release wheels as the current install
  path and keeps PyPI out of the default install promise.

## Release Check Script Design

Add `scripts/release_check.py` as a maintainer-facing pre-release verifier.

Expected capabilities:

- `--version 0.1.5` is required.
- `--tag v0.1.5` is optional and checks only expected tag spelling; this slice
  must not require that the tag already exists.
- Read `pyproject.toml` with `tomllib` and compare `project.version`.
- Read `lab_sidecar/__init__.py` with a small AST-based assignment reader and
  compare `__version__`.
- Confirm `CHANGELOG.md` contains a heading for the target version.
- Confirm `git diff --check` passes.
- Confirm the repository root does not contain `.lab-sidecar/`.
- If `dist/` exists, require exactly target-version wheel/sdist names and
  reject stale `lab_sidecar-0.1.4...` style artifacts.
- Inspect wheel `METADATA` and sdist `PKG-INFO` to confirm the published
  artifact contents declare `Name: lab-sidecar` and the target version, not
  only target-looking filenames.
- Check whitespace in unstaged diffs, staged diffs, and committed `HEAD` so
  pre-tag checks can catch errors before and after release-candidate changes
  are committed.
- Report SHA-256 digests for target-version distribution artifacts.
- Emit clear, actionable failure messages and exit non-zero on failures.
- Never publish, upload, tag, or mutate release state.

## GitHub Release Wheel Install Smoke Design

Keep `scripts/wheel_smoke.py` as the repository-build smoke. Add a separate
asset/install smoke so release operators can validate a distribution artifact
that already exists outside the current build.

Expected behavior for `scripts/release_asset_smoke.py`:

- Accept `--wheel <path-or-url>` and `--workspace <path>`.
- Default to local file paths and `file://` URLs; HTTP(S) URLs are explicit
  manual release checks and not used by default tests.
- Create an isolated virtual environment.
- Install the supplied wheel.
- Run `labsidecar --help`, `labsidecar doctor`, and a compact local
  `init -> run -> collect -> validate` path from checked-in examples.
- Print a JSON summary with wheel path/URL, installed version, installed CLI
  path, task id, command results, and status.
- Support `--version <version>` so a valid older wheel cannot silently satisfy
  the release-asset smoke for a newer release.
- Refuse to clear arbitrary non-smoke workspaces.

Tests should cover parsing, workspace safety, local wheel install flow with
mocked subprocess calls, and script `--help`. Network behavior remains manual.

## Artifact Contract Golden Fixture Strategy

Existing contract tests generate real outputs and validate public shapes. v0.1.5
adds small checked-in golden fixtures to make critical artifact contracts more
reviewable without committing full task directories.

Fixture rules:

- Store fixtures under `tests/fixtures/artifact_contract/v1/`.
- Keep each fixture small and hand-curated.
- Include only public contract JSON files:
  `manifest.json`, `metrics/collection-summary.json`,
  `provenance/traceability.json`, `package-summary.json`, and
  `artifact-index.json`.
- Use sanitized relative paths and placeholder task ids.
- Include required fields, `schema_version`, hash/size fields, run-mode
  compatibility, and explicit omission notes.
- Do not include full stdout/stderr logs, raw source bodies, full metric rows,
  worker prompt/response bodies, PPTX internals, local private paths, or an
  entire `.lab-sidecar/` output tree.

Tests should validate both generated outputs and the golden fixture set where
that keeps the contract compact and useful.

## Docs And Release Notes Strategy

Update public and maintainer-facing docs only where they improve release/install
trust:

- README: add a GitHub release wheel install path and explicit verification
  commands (`labsidecar --help`, `labsidecar doctor`).
- README and release checklist: keep editable install and wheel smoke paths,
  but distinguish repository development from release artifact installation.
- `docs/release-checklist.md`: add the release check script and release asset
  smoke to the pre-tag gate.
- `CHANGELOG.md`: add `0.1.5` with cautious public-alpha language.
- Add `docs/v0.1.5-install-release-trust-acceptance.md` with exact validation
  commands and observed results.
- Update `docs/cli-spec.md` only if command/install wording needs current
  console-script verification language.
- Avoid claiming PyPI availability, OS sandboxing, malware detection, hosted
  service behavior, or remote execution safety.

## Risks And Rollback

- Release check false positives can block a maintainer on harmless local state.
  Keep checks explicit, actionable, and limited to pre-release hygiene.
- Release check false negatives can still miss GitHub-side release body errors.
  Keep the script focused on local repository and artifact facts, then document
  manual GitHub release review separately.
- Network URL wheel smoke can be flaky. Keep URL mode manual and test only the
  local/mocked path by default.
- Golden fixtures can become brittle if they snapshot too much. Keep them
  minimal and contract-oriented rather than full generated outputs.
- Documentation can overclaim install availability. Keep PyPI out of default
  install promises until the package is actually published there.

Rollback is simple: revert this release-hardening slice before tagging. It does
not migrate user artifacts, change the task workflow, or introduce release
side effects.

## Explicit Release Boundary

This stage does not create a git tag, does not create or publish a GitHub
release, and does not publish to PyPI. It prepares a ready PR with release
trust checks and documentation for a maintainer-operated release flow.
